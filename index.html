Ruby Basics

Concept Usage Examples Description
Array []

[]

["Rails", "fun", 5]

 Creates an Array, empty or with the specified contents. An empty is a collection of things indexed with numbers.
Acces array an_array[number]

an_array = ["Helau", "Mooo", "zoooo"]

an_array[0] --> "Helau"

an_array[2] --> "zoooo"

 As an Array is a collection of different elements, you often want to access a single element of this Array. Arrays are indexed by numbers, you can use that number to access an individual element. Be aware that the numbering actually starts 0 so the first element actually is the 0th. And the last element of a 3 element array is element number 2.
Assigning Array Values an_array[number] = new_value

an_array = ["Helau", "Mooo", "zoooo"]

an_array[2] --> "Oh I'm all new"

an_array --> ["Helau", "Mooo", "Oh I'm all new"]

 Assigning new Array Values works a lot like accessing them, you just add an equals sign and a new value et voila you changed an element of the array! Weehuuuuu!

This will go into rails specific basics. Most prominently the general structure of a Rails application and the important command line commands.

The Structure of a Rails app

An overview of all the folders of brand new Rails application, outlining what they are there for. Also touching upon the most important files.
Name Description
app This folder contains your application. Therefore it is the most important folder in ruby on rails and it is worth digging into its sub folders. See the following rows.
app/assets Assets basically are your front end stuff. This folder contains images you use on your website, javascripts for all your fancy front end interaction and stylesheets for all your css making your website absolutely beautiful.
app/controllers The controllers subdirectory contains the different controllers of an application. A controller handles the requests from the users. It is often responsible for a single resource type, such as places, users or attendees. Controllers also tie together the models and the views.
app/helpers Helpers are used to take care of logic that is needed in the views in order to keep the views clean of logic and reuse that logic in multiple views.
app/mailers Functionality to send emails goes here.
app/models The models subdirectory holds the classes that model the business logic of our application. It is concerned with the things our application is about. Often this is data, that is also saved in the database. Examples here are a Person, or a Place class with all their typical behaviour.
app/views

The views subdirectory contains the display templates that will be displayed to the user after a successful request. By default they are written in HTML with embedded ruby (.html.erb). The embedded ruby is used to insert data from the application. It is then converted to HTML and send to the browser of the user. It has subdirectories for every resource of our application e.g. places, persons. These subdirectories contain the associated view files.

Files starting with an underscore (_) are so called partials. Those are parts of a view which are reused in other views. A common example is _form.html.erb which contains the basic form for a given resource. It is used in the new and in the edit view since creating something and editing something looks pretty similar.

config This directory contains the configuration files that your application will need, including your database configuration (in database.yml) and the particularly important routes.rb which decides how web requests are handled. More specifically: for a given URL it decides which controller will handle the request.
db Contains a lot of database related files. Most importantly the migrations subdirectory, containing all your database migrations. Those files basically define how your database should look like. This is important as it, among other things, defines what attributes your models have. With migrations you can add new attributes to existing models or create new ones. So you could add the favorite_color attribute to your Person model so everyone can specify their favorite color.
doc Contains the documentation you create for your application. Not too important when starting out.
lib Short for library. Contains code you developed that is not only specific to your application but could be used elsewhere. This might be a set of functionality related to getting specific information from Facebook.
log See all the funny stuff that is written in the console where you started the rails server? That goes here into development.log. Basically logs contain runtime information of your application. Also if an error happened, it will be noted here.
public Contains general non dynamic files like some html pages.
script By default contains what is executed when you type in the rails command. That's not really important to know in the beginning though.
test Contains the tests for your application. With tests you make sure that your application actually does what you think it does. This directory might also be called spec, when you are using rspec/
vendor Libraries that you do not own e.g. somebody else owns them. You will install most of those as ruby gems (see Gemfile) and those will not show up here. However if they are not available as ruby gems then you should put them here. This might be the case for jQuery plugins. Probably won't be used that often in the beginning.
Gemfile

A list of the gems, more generally called libraries, (= set of functionality) your application depends on. For Rails applications this will always be rails.

If you want to add a new gem to your app all you have to do is add "gem gem_name" to your Gemfile, optionally specifying a version number. Save the file and then run bundle install to install the gem.

Gemfile.lock This file contains the versions of all gems you use. Because some gems depend on other gems so ruby is so nice to install them for you automatically. It also contains the concrete version numbers. It can be used to make sure that everyone within a team is working with the same versions of gems. The file is auto generated. Do not touch this file.

What to do when things go wrong?

Things go wrong all the time, so don't worry this happens to everyone. So keep calm. When you encounter an error message it is always a good idea to google the error message and maybe at key words like "rails" or "ruby".

There are some mistakes, which I found to be common when an error occurs so I made this little checklist for you here:

  • Have you run rake db:migrate to apply the newest database migrations?
  • Have you really saved the file you just changed? Unsaved files are often marked in the editor via an asterisk or a point next to their name.
  • If you just added a gem to the Gemfile, have you run bundle install to install it?
  • If you just installed a gem, have you restarted the server?
